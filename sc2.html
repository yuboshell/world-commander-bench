World Commander — StarCraft II testbed

Updated: 2026-06-21 · Machine: yubopc (RTX 4060, 8 GB) · Game: SC2 5.0.15 · Harness: LLM-PySC2 (patched) · Model: Qwen3-4B-AWQ via vLLM (WSL2, localhost:8001) · members-only

TL;DR. A small LLM (Qwen3-4B) issues natural-language battle commands to StarCraft II units through LLM-PySC2. The key result, from a controlled study on the balanced 3s5z matchup: the 4B is capable enough to roughly match the game's built-in auto-attack (≈77% vs ≈67% win rate, +10 pp, not statistically significant) — but it is ~25× too slow to act in real time (~27 s per decision). On this testbed the binding constraint is latency, not capability.

1 · Watch a game — and the language→game link

A complete 3s5z game, rendered top-down so every unit is visible — allies in cyan, enemy in red, each sized by current health, the live count falling as units die. Captions name each phase and show the actual LLM command behind it (focus-firing the enemy Stalkers; Blink and Charge micro). The model commands the cyan army and, in this game, is out-microed and loses. Sped up — each real decision takes ~27 s.

What actually happens between a command and the game

Each decision, the harness sends the LLM the full game state (every unit's position, health, type — ~2.4–5.7 K tokens of text). The model replies with a short natural-language analysis plus structured actions in a small command language (<Attack_Unit(tag)>, <Move_Screen([x,y])>, <Charge()>, <Blink(...)>). LLM-PySC2 parses those into pysc2 actions (select unit → issue ability → click target on screen) and steps the game. Two real decisions from the clip above:

Decision 1

"The closest enemy Zealots are at distances 1–2… focus on attacking the closest enemy Zealots first… Stalkers can use their Blink ability to reposition; Zealots use Charge to close the distance."

Team Zealot-1: <Attack_Unit(0x100340001)> <Charge()>

Decision 2 — concentrate fire

"The closest enemy Zealot (Tag 0x100340001) is at distance 1, an ideal target… focus on the closest enemy Zealot first."

Zealot-1 / Zealot-2 / Zealot-3 → all <Attack_Unit(0x100340001)>

The reasoning is sensible RTS micro — focus-fire the nearest/weakest enemy, use abilities — and it maps cleanly onto game actions. The catch (below) is that "attack the nearest" is largely what the units already do on their own.

2 · Capability — does the LLM actually help?

SC2 units auto-attack on their own, so a raw win doesn't prove the model did anything. We control with a no-LLM baseline (WCB_SC2_MAX_QUERIES=0) and pool the high-n runs by condition:

conditionwin-raten
LLM acts (model in the loop)49/64 ≈ 77%64
No effective LLM (auto-attack / dropped)32/48 ≈ 67%48
Difference +10 pp, two-proportion p≈0.25 — not significant; the gap shrank as n grew (+22→+17→+10 pp).

And win-rate tracks the force matchup, not the model:

matchupauto-attackwith LLMread
1c3s5z (overwhelming)100%100%auto-wins; LLM irrelevant
3s5z (balanced)~67%~77%LLM helps modestly (n.s.)
2s3z / 3s_vs_3z (unwinnable)0%0%LLM neutral

Why the effect is small (from reading the commands): the 4B's micro is reasonable but largely redundant with auto-attack — it focus-fires the nearest/weakest enemy, which is ~what the units already do. So it mostly commands what would happen anyway. It's redundancy, not incompetence; a bigger edge needs decisions that diverge from the default (true kiting, cross-team focus) more reliably than this model produces.

3 · The clock — latency is the binding constraint

Per-decision cost on 3s5z: ~27 s/decision (range 25.5–30.1 s), prefill-dominated — the prompt is 2.4–5.7 K input tokens (the full unit-by-unit observation), output only ~150–315 tokens. So latency scales with the observation size, not the generation.

At ~27 s/decision the 4B is ~25× too slow for second-scale RTS control (≈5× too slow even versus a generous 5 s deadline). LLM-PySC2's default deadline is soft — a late reply is applied anyway — so win-rate is deadline-invariant. We added a true real-time drop-late clock (WCB_SC2_DROP_LATE=1): a reply that misses its deadline is discarded. Validated at a 10 s deadline: 13/13 late replies dropped, attack actions 45→1, and the agent reverts to the auto-attack baseline. That is the program's thesis made concrete: under a real clock, a too-slow model contributes nothing.

The lever for real-time viability is therefore not just a smaller model — it's shrinking the observation prompt (prefill is the cost) and KV-cache reuse across ticks. That is the efficiency sweep the program scopes next.

4 · Methods, reliability & caveats

  • Auto-attack confound controlled with no-LLM runs; win/loss read from the obs-list-episode<N>-{win,lose,tie}.pkl suffix.
  • Camera-calibration bootstrap was run-fragile (some runs never queried the LLM). Fixed for reliability with a world_range guard + an env-gated force-proceed cap (WCB_SC2_CALIB_CAP); ~75% of runs now calibrate cleanly. A full "every run converges" fix is in progress.
  • Single model (4B); 8 GB GPU caps model size. Binomial CIs are wide; the pooled headline uses n≈48–64/side.
  • Full data, code patch, and the evolving analysis trail: results/sc2/sweep_2s3z_4B.md in the repo.
